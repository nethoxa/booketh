Upgradeable

DEEP

The key here is that whenever a contract A delegates a call to another contract B, it executes the code of contract B in the context of contract A. This means that msg.value and msg.sender values will be kept and every storage modification will impact the storage of contract A

Master-Slave

The master holds the addresses of its slaves, and they request the addresses of the other slaves from the master. Thus, we deploy a slave and change its relative address in the master contract

Eternal Storage contracts

The logic and data contracts are separated from each other. The data contract is supposed to be permanent and non-upgradable and the logic one can be upgraded as many times as needed. However, one bug in the data contract and all the data becomes useless. Master-Slave is better

Proxies

Just deploy a contract which delegatecalls the logic and storage contracts

Unstructured Proxies

It works by saving the addresses of the implementation and the owner at fixed positions in the storage such that they won’t be overwritten by the data. We can use the sload and sstore opcodes to directly read and write to specific storage slots. If we set the fixed position to something like 0x7 then it will get overwritten just after first 7 storage slots are used. To avoid this, we set the fixed storage position to something like keccak256(...) to avoid collisions

Suppose that the proxy stores the logic contract’s address in its only variable address public _implementation;. Now, suppose that the logic contract is a basic token whose first variable is address public _owner. Both variables are 32 byte in size, and as far as the EVM knows, occupy the first slot of the resulting execution flow of a proxied call. When the logic contract writes to _owner, it does so in the scope of the proxy’s state, and in reality writes to _implementation. This problem can be referred to as a "storage collision".
