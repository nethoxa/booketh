Stack

The EVM is a stack-based VM. That means all the basic computations are made on a stack 32 byte (256 bits) wide and 1024 elements in size. There are a few things to take into account:

  • It is the cheapest to use data location
  • It's only avaiable in the current function scope (or in the inline blocks {...} inside them)
  • There are four basic operations (opcodes):
    • PUSH
    • POP
    • SWAP (not in contracts)
    • DUP (not in contracts)
  • Other opcodes always take the topmost items on the stack

The working process of the stack is that an operation consumes n elements of the stack and PUSHes result on top of it. For that reason, if an operation requires n elements but there are m avaiable such as n < m, then there is an underflow and the EVM halts its execution. The same applies if we start PUSHing more than the size of the stack. However, there is something weird. You can only access the first 16 elements at a time. You can access others which are deeper on the stack, but the EVM will POP the one above the desired one. Whatever, for our purpose, we don't give a fuck about that (yet)

One more thing is that variables smaller than 32 bytes are padded, which is pretty interesting. Depending on the type, we have two options:

  • bytesN are right padded
  • address and uint are left padded

We talked before about inline blocks. It matters because the stack will maintain in the innermost block the variables of the outmost one. However, once it ends, all the variables INSIDE the innermost block are popped, thus making them innaccesible form the outmost one. It's just the same as C, Rust or whatever language you like

There is another stack, apart from the data one, which is the "call" one. In short, it pushes the address of the current contract and those which are called. For example, suppose we have A which calls B and then returns. Then

  1. First, the call stack has the address of A
  2. Once it calls B, its address gets PUSHed
  3. At the end, its address is POPped and the executions continues in A until it finishes, getting POPed too

Memory

You can think of memory as an array of byte[]. You can read and write to it, not as calldata, which is just read-only. First, you need to know something (photo from Jean Cvllr):

  • From 0x00 to 0x3F -> scratch space for hashing functions
  • From 0x40 to 0x5F -> FREE MEMORY POINTER
  • From 0x60 to 0x7F -> meant to be zero permanently and used as the initial value for empty dynamic memory arrays

If you put an offset higher than 2^64 - 1, the transaction will revert. The following data and values are always in memory by default:

  • Function parameters of complex types
  • Local variables of complex types
  • Values returned from functions, regardless of their type
  • Any complex value type returned by a function must specify the keyword memory
  • By variables/values of complex types, we refer to variables such as struct, arrays, bytes and strings

These variables defined with the keyword memory will disappear once the function call ends. For reads from memory, we are limited to 32 bytes words. However, we can use either 1 byte or 32 bytes words in writes. The reason underneath is due to the opcodes MLOAD to read (32 bytes), MSTORE and MSTORE8 to write (32 bytes and 8 bytes respectively).

For MLOAD

the formula is explained as follows (again, from Jean Cvllr):

  • Us[0] is the top item on the stack
  • Us'[0] is the return item PUSHed on the stack
  • Um is the *v[i], that is, the content of memory at a given offset i
  1. Take the last top item on the stack Us[0]
  2. Use this value as the starting pointer to read in memory Um
  3. Read from this memory pointer Us[0] the next 31 bytes that follow (Us[0] + 31)

We see now that we can only read from memory 32 bytes words at a time with a single MLOAD opcode.

Calldata

When calling a contract, the function's parameters are stored in a special storage place called calldata. In short, it's the memory of the caller, but made of bytes laid out continuously instead of 32 bytes words and with the distinction of being READ ONLY.

The first 4 bytes is the signature of the called function, and the remaining bytes correspond to the input arguments PADDED AS IN THE STACK.

One thing to know is that allocating more bytes in the calldata is always linear in comparison to the memory that goes quadratically as the memory size grows. Thus, it's more gas efficient to store stuff in the calldata rather than in memory.

Just a few things to take into account:

  • msg.sender -> caller
  • msg.value -> value, amount, IDK, the ETH in selfdestruct attacks xD
  • msg.sig -> the first 4 bytes of the calldata (see above)
  • msg.data -> all de calldata, including the signature
  • gasleft() -> the remaining gas available

About msg.sig, the 4 bytes are from doing keccak256("FUNCTION_NAME(TYPE1,TYPE2,...)). See that THERE IS NO SPACE BETWEEN TYPES, so if there is typo in there it will change the whole hash.

There are a few tings to build the calldata:

  • As a literal string -> hardcoded like "0xagshdvaqjklmwdeqoedryqkajhgeuye1" (it's not hex, but you know the point)
  • The same as above but writing hex"agshdvaqjklmwdeqoedryqkajhgeuye1" without the 0x
  • Hash the function and append the parameters at the end and do stuff

    /*
                                INTERNAL
    */

        _mint(to, tokenId);
        require(
            _checkOnERC721Received(address(0), to, tokenId, data),
            "ERC721: transfer to non ERC721Receiver implementer"
        );
    }

   
    function _mint(address to, uint256 tokenId) internal virtual {
        require(to != address(0), "ERC721: mint to the zero address");
        require(!_exists(tokenId), "ERC721: token already minted");

        _beforeTokenTransfer(address(0), to, tokenId, 1);

        // Check that tokenId was not minted by `_beforeTokenTransfer` hook
        require(!_exists(tokenId), "ERC721: token already minted");

        unchecked {
            // Will not overflow unless all 2**256 token ids are minted to the same owner.
            // Given that tokens are minted one by one, it is impossible in practice that
            // this ever happens. Might change if we allow batch minting.
            // The ERC fails to describe this case.
            _balances[to] += 1;
        }

        _owners[tokenId] = to;

        emit Transfer(address(0), to, tokenId);

        _afterTokenTransfer(address(0), to, tokenId, 1);
    }

    
    function _burn(uint256 tokenId) internal virtual {
        address owner = ownerOf(tokenId);

        _beforeTokenTransfer(owner, address(0), tokenId, 1);

        // Update ownership in case tokenId was transferred by `_beforeTokenTransfer` hook
        owner = ownerOf(tokenId);

        // Clear approvals
        delete _tokenApprovals[tokenId];

        // Decrease balance with checked arithmetic, because an `ownerOf` override may
        // invalidate the assumption that `_balances[from] >= 1`.
        _balances[owner] -= 1;

        delete _owners[tokenId];

        emit Transfer(owner, address(0), tokenId);

        _afterTokenTransfer(owner, address(0), tokenId, 1);
    }

    
    function _transfer(address from, address to, uint256 tokenId) internal virtual {
        require(ownerOf(tokenId) == from, "ERC721: transfer from incorrect owner");
        require(to != address(0), "ERC721: transfer to the zero address");

        _beforeTokenTransfer(from, to, tokenId, 1);

        // Check that tokenId was not transferred by `_beforeTokenTransfer` hook
        require(ownerOf(tokenId) == from, "ERC721: transfer from incorrect owner");

        // Clear approvals from the previous owner
        delete _tokenApprovals[tokenId];

        // Decrease balance with checked arithmetic, because an `ownerOf` override may
        // invalidate the assumption that `_balances[from] >= 1`.
        _balances[from] -= 1;

        unchecked {
            // `_balances[to]` could overflow in the conditions described in `_mint`. That would require
            // all 2**256 token ids to be minted, which in practice is impossible.
            _balances[to] += 1;
        }

        _owners[tokenId] = to;

        emit Transfer(from, to, tokenId);

        _afterTokenTransfer(from, to, tokenId, 1);
    }

    
    function _approve(address to, uint256 tokenId) internal virtual {
        _tokenApprovals[tokenId] = to;
        emit Approval(ownerOf(tokenId), to, tokenId);
    }
}

abstract contract ERC4626 is ERC20 {
    /*
                                PUBLIC
    */
    function deposit(uint256 assets, address receiver) public virtual returns (uint256 shares) {
        // Check for rounding error since we round down in previewDeposit.
        require((shares = previewDeposit(assets)) != 0, "ZERO_SHARES");

        // Need to transfer before minting or ERC777s could reenter.
        asset.safeTransferFrom(msg.sender, address(this), assets);

        _mint(receiver, shares);

        emit Deposit(msg.sender, receiver, assets, shares);

        afterDeposit(assets, shares);
    }

    
    function mint(uint256 shares, address receiver) public virtual returns (uint256 assets) {
        assets = previewMint(shares); // No need to check for rounding error, previewMint rounds up.

        // Need to transfer before minting or ERC777s could reenter.
        asset.safeTransferFrom(msg.sender, address(this), assets);

        _mint(receiver, shares);

        emit Deposit(msg.sender, receiver, assets, shares);

        afterDeposit(assets, shares);
    }

    
    function withdraw(
        uint256 assets,
        address receiver,
        address owner
    ) public virtual returns (uint256 shares) {
        shares = previewWithdraw(assets); // No need to check for rounding error, previewWithdraw rounds up.

        if (msg.sender != owner) {
            uint256 allowed = allowance[owner][msg.sender]; // Saves gas for limited approvals.

            if (allowed != type(uint256).max) allowance[owner][msg.sender] = allowed - shares;
        }

        beforeWithdraw(assets, shares);

        _burn(owner, shares);

        emit Withdraw(msg.sender, receiver, owner, assets, shares);

        asset.safeTransfer(receiver, assets);
    }

    
    function redeem(
        uint256 shares,
        address receiver,
        address owner
    ) public virtual returns (uint256 assets) {
        if (msg.sender != owner) {
            uint256 allowed = allowance[owner][msg.sender]; // Saves gas for limited approvals.

            if (allowed != type(uint256).max) allowance[owner][msg.sender] = allowed - shares;
        }

        // Check for rounding error since we round down in previewRedeem.
        require((assets = previewRedeem(shares)) != 0, "ZERO_ASSETS");

        beforeWithdraw(assets, shares);

        _burn(owner, shares);

        emit Withdraw(msg.sender, receiver, owner, assets, shares);

        asset.safeTransfer(receiver, assets);
    }

    
    function convertToShares(uint256 assets) public view virtual returns (uint256) {
        uint256 supply = totalSupply; // Saves an extra SLOAD if totalSupply is non-zero.

        return supply == 0 ? assets : assets.mulDivDown(supply, totalAssets());
    }

    function convertToAssets(uint256 shares) public view virtual returns (uint256) {
        uint256 supply = totalSupply; // Saves an extra SLOAD if totalSupply is non-zero.

        return supply == 0 ? shares : shares.mulDivDown(totalAssets(), supply);
    }
}

Let's see the new parameters:

  • One-to-one relation -> assets[i] ~ amounts[i] ~ modes[i]
  • assets -> assets being flash-borrowed
  • amounts -> amounts being flash-borrowed
  • modes -> types of the debt to open if the flash loan is not returned:

    • 2 -> Open debt at variable rate for the value of the amount flash-borrowed to the onBehalfOf address
  • params -> packed params to pass to the receiver as extra information

        case 0 { result := 1 } // something^0 is 1
        case 1 { result := base } // something^1 is something
        default {
            result := power(mul(base, base), div(exponent, 2)) // recursive call with sqrt(exponent)

Storage slots

To get the storage slot of a variable, use VAR.slot which is the POINTER TO SOMEWHERE, NOT THE VALUE THERE. To read/write to it, do sload(VAR.slot) or sstore(VAR.slot, VAR_EXTERNAL). Next is an example of multiple variables being packed into one slot (from here):

uint128 a;
uint96 b;
uint16 c;
uint8 d;

function set(uint16 _c) public {
    assembly{
        // Get the storage slot of the variable
        let wholeSlot := sload(c.slot)

        // Clear the variable's bits in the slot. Since it is a uint16, it is 2 bytes long.
        let cleared := and(wholeSlot, 0xffff0000ffffffffffffffffffffffffffffffffffffffffffffffffffffffff)

        // Shift the new value to the left by the offset of the variable multiplied by 8(1 byte = 8 bits)
        let shifted := shl(mul(c.offset, 8), _c)

        // Combine the cleared slot and the shifted value
        let newValue := or(shifted, cleared)

        // Store the new value in the slot
        sstore(c.slot, newValue)
    }
}

function get() public view returns (uint16 c_) {
    assembly {
        // Get the storage slot of the variable
        let wholeSlot := sload(c.slot)

        // Shift the slot to the right by the offset of the variable
        let shifted := shr(mul(c.offset, 8), wholeSlot)

        // Mask the slot to get the value of the variable
        c_ := and(shifted, 0xffff)
    }
}

Unicode

It's not exactly Yul but there is a Unicode character that can be used to force a right-to-left direction withing a text:

my-text.'U+202E'cod.exe -> my-text.exe.doc instead of my-text.doc.exe

To type it in Linux do:

  1. Hold ctrl + shift
  2. Type U202E
  3. Release ctrl + shift

Use that whenever you can to break things that accept string or bytes. To put them in Yul, just do \u202E. From the docs:

An ASCII string is first viewed as a byte sequence, by viewing a non-escape ASCII character as a single byte whose value is the ASCII code, an escape \xNN as single byte with that value, and an escape \uNNNN as the UTF-8 sequence of bytes for that code point. The byte sequence must not exceed 32 bytes. The byte sequence is padded with zeros on the right to reach 32 bytes in length; in other words, the string is stored left-aligned. The padded byte sequence represents a 256-bit word whose most significant 8 bits are the ones from the first byte, i.e. the bytes are interpreted in big endian form

So that we could put in an argument that needs a type less than 32 bytes something like VALUE\u202E which will be padded until 32 bytes and once it gets processed, it will be converted to 000...VALUE (kinda, I have to test it)

Dynamic arrays and mappings

About dynamic arrays, the elements are located not in the slot of the v but rather on keccak256(slot) + (index + element_size) with a cast to uint256. For mappings, they are located on keccak(key, slot) with a cast to uint256. For nested types, we do a from-the-ouside approach, calculating the hashes of the outermost structure first. For example:

// dynamic arrays

assembly {
    slot := VAR.slot
}

bytes32 start = keccak256(abi.encode(slot))
uint256 sol;

// v[start + target]
assembly {
    sol := sload(add(start, target))
}

// nested mappings => mapping(uint256 => mapping(uint256 => uint256))
uint256 slot;

assembly {
    slot := VAR.slot
}

// hashs the key and uint256 value of slot
bytes32 locationOfParentValue = keccak256(abi.encode(key1, slot));
// hashs the parent key with the nested key
bytes32 locationOfNestedValue = keccak256(abi.encode(key2, locationOfParentValue));

uint256 sol;
// loads storage slot of location and returns ans
assembly {
    sol := sload(locationOfNestedValue)

Packed values

To read/write packed values we must do the following:

  1. Load the storage slot -> slot := VAR.slot
  2. Create a mask to isolate the value -> 0x000...FFFFF or whatever, just to clear the value we do not need
  3. let VAR2 := and(slot, mask) -> read here, continue if write
  4. let SHIFTEDVAR := shl/shr(mul(VAR.offset, 8), NEW) -> the mult is because VAR.offset is in bytes, and we have to convert that to bits. It's read like shift to the left/right VAR.offset * 8 bytes the variable/slot NEW
  5. let NEWSLOT := or/and(SHIFTEDVAL, VAR2)

Just know that variables will always be packed from down to up. So this:

uint128 public C = 4;
uint104 public D = 6;
uint16 public E = 8;
uint8 public F = 1;

will be packed like this:

0x0100080000000000000000000000000600000000000000000000000000000004

Another example of reading from storage:

assembly{
    let slot := sload(VAR.slot)
    VAR := shr(mul(VAR.offset, 8), slot)
    let sol := and(0xffff,VAR)
}

For writing:

assembly{
    // lets overwrite una var in VAR
    value:= sload(VAR.slot)
    // if we know where is located, we just reset it with an and like this => the 0's are the places where is packed var 
    cleared := and(0xff0000ffffffffffffffffffffffffffffffffffffffffffffffffffffffffff, value)
    // we "move" the variable to write to its position
    shifted := shl(mul(var.offset,8), NEW)
      
    // write
    NEWV := or(shifted,cleared)
    sstore(VAR.slot, NEWV)
}

It's just a three step process:

  1. Mask
  2. Move (shl/shr)
  3. Read/Write (and/or)

Tips

  • Change lt(i, n) with iszero(eq(i, n)) because it saves 4 gas/iteration
  • Change mul(2^n, value) with shl(2^n, value). The same with div and shr
  • Replace for with for {} 1 {} {...} with a break around there
  • Changing switch-case → infinite for loop with if-break statements, saves 1 gas + more when eq() → iszero(gt())
  • Solidity auto-cleans upper-bits of down-casted variables while executing comparison operators. In Yul, we need to do it manually.
