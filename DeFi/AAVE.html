Introito

Just know that AAVE is THE lending pool (borrowers take and depositors provide liquidity in exchange for "fees"). The entry point for us (and everyone else) is the LendingPool which offers the next functions to interact with (the most interesting):

  • deposit -> Deposits an amount of tokens into the reserve, receiving in return aTokens (AAVE tokens). That is, you deposit 100 USDC and get in return 100 aUSDC
    • Parameters
      • address asset -> the token address
      • uint256 amount -> self-explanatory (nothing from now on)
      • address onBehalfOf -> who does receive the aTokens
      • uint16 referralCode -> usually 0
  • withdraw -> The inverse of deposit, burning the aTokens and getting in return the tokens
    • Parameters
      • address asset
      • uint256 amount
      • address to
  • borrow -> Borrows amount of asset with interestRateMode, sending the amount to msg.sender, with the debt being incurred by onBehalfOf
    • Parameters
      • address asset
      • uint256 amount
      • uint256 interestRateMode -> type of interest you will pay for your debt (think of college debt being stable, variable... not if you are from Europe though xD)
      • uint16 referralCode
      • address onBehalfOf
  • repay -> Self-explanatory, just return the borrowed debt plus interests
    • Parameters
      • address asset
      • uint256 amount
      • uint256 rateMode
      • address onBehalfOf
  • swapBorrowRateMode -> Change the debt from stable to variable or whatever. The function changes the one FROM msg.sender
    • Parameters
      • address asset
      • uint256 rateMode
  • setUserUseReserveAsCollateral -> Sets the asset of msg.sender to be used as collateral or not
    • Parameters
      • address asset
      • bool useAsCollateral
  • liquidationCall -> The caller (liquidator) covers debtToCover amount of debt of the user getting liquidated, and receives a proportionally amount of the collateralAsset plus a bonus to cover market risk (sharks)
    • Parameters
      • address collateralAsset -> underlying asset used as collateral to receive as result of the liquidation
      • address debtAsset -> underlying borrowed asset to be repaid with the liquidation
      • address user -> user getting liquidated
      • uint256 debtToCover -> amount of borrowed asset the liquidator wants to cover
      • bool receiveAToken -> true if the user wants the aTokens, false if he wants the tokens

If you wanna know more about the inner workings of AAVE, just read the whitepaper (14 pages, you can read it even while *******)

Flashloans

Now we will talk about what we all wanna know, flashloans. The function responsible for that is:

function flashLoanSimple(
    address receiverAddress,
    address asset,
    uint256 amount,
    bytes calldata params,
    uint16 referralCode
);

function flashLoan(
    address receiverAddress,
    address[] calldata assets,
    uint256[] calldata amounts,
    uint256[] calldata modes,
    address onBehalfOf,
    bytes calldata params,
    uint16 referralCode
);

Let's see the new parameters:

  • receiverAddress -> the contract receiving the funds implementing the IFlashLoanReceiver interface
  • One-to-one relation -> assets[i] ~ amounts[i] ~ modes[i]
  • assets -> assets being flash-borrowed
  • amounts -> amounts being flash-borrowed
  • modes -> types of the debt to open if the flash loan is not returned:
    • 0 -> Don't open any debt, just revert if funds can't be transferred from the receiver
    • 1 -> Open debt at stable rate for the value of the amount flash-borrowed to the onBehalfOf address
    • 2 -> Open debt at variable rate for the value of the amount flash-borrowed to the onBehalfOf address
  • params -> packed params to pass to the receiver as extra information

Ok, but why the fuck do I need to know that? Well, this is a process, so we need to start from the basics, but you can go straight to the example.

Execution path

From the docs, the execution path is as follows (simplified):

  1. YOUR CONTRACT calls flashLoanSimple() or flashLoan()
  2. YOUR CONTRACT receives the money and YOUR executeOperation() is called
    1. If flashLoanSimple, then when your code has finished, you approve Pool for flash loaned amount + fee.
    2. If flashLoan, then for all the reserves either depending on interestRateMode passed for the asset, either the Pool must be approved for flash loaned amount + fee or there must be enough collateral or credit delegation should be available to open debt position
  3. If the amount owing is not available (due to a lack of balance or approval or insufficient collateral for debt), then the transaction is reverted. All of the above happens in 1 transaction (single ethereum block)
  4. NOTE -> You do not need to transfer the owed amount back AAVE, he will take it back automatically at the end of your operation (nice)

A simplified diagram of a successful flashloan (without opening debt positions nor multi-assets loans) is as follows

flowchart LR
    YOU(YOU) --1. flashLoanSimple/flashLoan--> AAVE(AAVE)
    AAVE --2. money--> YOU
    YOU --3. do something--> WORLD(WORLD)
    YOU --4. repay money + fee--> AAVE

The most important part (for us) is the 3. do something that interacts with the outer world. Here you will POC arbitrages, governance takeovers and all of that stuff, but first the example/template
